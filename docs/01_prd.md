# AI 新闻聚合网站 MVP PRD

## 产品目标

帮助用户及时了解最新、最有价值的 AI 资讯。系统通过 RSS 信息源抓取新闻，使用 LLM 进行新闻价值评分，仅对高价值新闻抓取全文，并尽快生成中文翻译，最终在前端展示新闻卡片、30 天高分榜单和中文全文阅读页。本文中的“新闻发布时间”均指 RSS 信息源给出的原始时间；本站只做抓取、整理和展示。

## 技术栈

- 前端：React + Vite 单页应用
- 后端：Python FastAPI
- 数据库：SQLite
- 定时任务：后端定时调度器
- 内容抓取：RSS/XML 解析器、HTML 正文抽取器
- LLM：用于新闻价值评分、中文翻译，所有调用必须使用结构化 JSON 输入输出

## 功能 1：RSS 信息源管理

### 闭环流程 1.1：初始化和新增 RSS 信息源

**输入**

- RSS 订阅链接或中转 RSS 链接
- 信息源名称
- 系统默认信息源列表：
  - `https://developers.openai.com/rss.xml`
  - `https://openai.com/news/rss.xml`
  - `https://dreyx.com/digest/rss`
  - `https://news.ycombinator.com/rss`
  - `https://hnrss.org/frontpage`
  - `https://hnrss.org/newest`
  - `https://hnrss.org/bestcomments`

**输出**

- 新增后的信息源记录
- 可统一管理的信息源列表

**运行流程**

1. 如果用户尚未配置过信息源，系统预装默认 RSS 信息源。
2. 默认 RSS 信息源和用户新增信息源进入同一张信息源列表统一管理。
3. 主页提供 `[配置]` 按钮，点击进入信息源配置页面。
4. 配置页面显示所有信息源，并支持新增和删除。
5. 用户输入 RSS 链接和信息源名称。
6. 后端校验链接格式必须为公开 `http/https` URL，且必须拒绝本地地址、私有地址和非 `http/https` URL。
7. 后端保存信息源 URL、名称、创建时间和启用状态。
8. 前端刷新信息源列表。

**验收标准**

- 输入合法 RSS URL 后，信息源列表出现该信息源。
- 输入非法 URL、本地地址或空 URL 时，接口返回错误，数据库不新增记录。
- 重复添加同一个 URL 时，不产生重复信息源记录；已软删除 URL 仍视为占用，除非未来提供重置配置流程。
- 用户未配置过信息源时，系统自动预装默认信息源。
- 预装信息源和用户新增信息源在配置页表现一致，均可删除。

### 闭环流程 1.2：查看和删除 RSS 信息源

**输入**

- 当前已保存的信息源列表
- 待删除的信息源 ID

**输出**

- 更新后的信息源列表

**运行流程**

1. 用户进入信息源配置页面。
2. 前端请求后端获取全部未删除的信息源。
3. 用户点击某个信息源的删除按钮。
4. 后端对该信息源执行内部软删除：写入删除事实，停止未来抓取，但不物理删除历史记录。
5. 前端重新加载信息源列表。

**验收标准**

- 配置页能展示所有未删除的信息源。
- 删除信息源后，该信息源不再出现在配置页。
- 删除信息源后，历史新闻记录仍保留。
- 禁用信息源仍可在配置页展示并重新启用；删除信息源不再展示，且不会被抓取。
- 预装信息源被删除后，不会在下一次启动或刷新时自动恢复，除非用户重置配置。

## 功能 2：RSS 定时抓取与新闻价值评分

### 闭环流程 2.1：每天定时抓取 RSS

**输入**

- 所有启用状态的信息源
- 系统配置的 `09:00` 和 `18:00` 抓取时间

**输出**

- RSS 原始新闻条目
- 抓取日志

**运行流程**

1. 后端调度器每天 `09:00` 触发一次抓取任务。
2. 后端调度器每天 `18:00` 触发一次抓取任务。
3. 系统逐个请求启用的信息源 RSS。
4. 系统解析 RSS 条目的标题、摘要、链接、信息源发布时间和来源。
5. 系统记录每个信息源的抓取成功或失败状态。

**验收标准**

- 每天 `09:00` 和 `18:00` 各执行一次抓取任务。
- 单个信息源抓取失败时，不影响其他信息源抓取。
- 抓取完成后能通过内部聚合、结构化测试报告或 pipeline-owned records 查询到本次任务的开始时间、结束时间、成功数和失败信息。
- 抓取任务摘要不通过 public API 暴露，不新增任务状态、进度或处理日志接口。

### 闭环流程 2.2：手动刷新抓取 RSS

**输入**

- 用户点击主页面 `[刷新]` 按钮
- 所有启用状态的信息源

**输出**

- 本次抓取结果
- 刷新后的主页面新闻列表

**运行流程**

1. 用户点击主页面 `[刷新]` 按钮。
2. 前端调用后端手动抓取接口。
3. 后端立即执行一次完整抓取任务。
4. 后端完成 RSS 抓取、评分、去重、全文抓取、设为展示就绪和翻译任务入队。
5. 后端返回本次任务结果。
6. 前端重新请求主页面新闻列表并刷新页面内容。

**验收标准**

- 点击 `[刷新]` 后，后端立即创建一条抓取任务记录。
- 抓取任务完成后，前端新闻列表重新加载。
- 刷新过程中按钮进入加载态，任务完成后恢复可点击。
- 单个信息源抓取失败时，页面仍展示其他信息源的最新内容。

### 闭环流程 2.3：基于标题和摘要进行 LLM 新闻价值评分

**输入**

- RSS 标题
- RSS 摘要
- 新闻原文链接
- 新闻来源
- 信息源发布时间

**输出**

- 新闻价值评分，范围为 `0-100`
- 是否进入全文抓取队列

**运行流程**

1. 系统读取新抓取的 RSS 条目，并将数据库 `pipeline_state` 标记为 `raw`。
2. 系统按标准 JSON 格式将标题、摘要、来源、信息源发布时间和原文链接发送给 LLM。
3. LLM 按标准 JSON 格式返回 `0-100` 的新闻价值评分。
4. 系统保存评分，并将数据库 `pipeline_state` 更新为 `scored`。
5. 评分大于或等于 `60` 的条目写入 `is_selected = 1`，进入全文抓取候选集合。
6. 评分小于 `60` 的条目写入 `is_selected = 0`，不抓取全文。
7. 标题或原文链接缺少任意一个，直接评分为 `0`。
8. 缺少摘要时，评分扣 `20` 分。

**验收标准**

- 每条新 RSS 条目都有一个数值评分。
- 评分大于或等于 `60` 的条目被标记为待抓取全文。
- 评分小于 `60` 的条目不会触发全文抓取。
- 标题或原文链接缺失的条目评分为 `0`，且不会触发全文抓取。
- 评分成功后，数据库 `pipeline_state` 必须从 `raw` 更新为 `scored`。
- 高价值新闻必须写入 `is_selected = 1`。

## 功能 3：高价值新闻全文抓取

### 闭环流程 3.1：高价值新闻去重

**输入**

- 评分大于或等于 `60` 的 RSS 条目

**输出**

- 去重后的待抓取新闻列表

**运行流程**

1. 系统读取评分大于或等于 `60` 的条目。
2. 系统根据原文链接生成规范化后的 `canonical_url`，并优先使用 `canonical_url` 作为去重键。
3. 如果没有原文链接，当前条目不得进入全文抓取候选集合。
4. 系统检查数据库是否已存在相同 `canonical_url`。
5. 已存在的条目不重复抓取全文。
6. 不存在的条目进入全文抓取候选集合。

**验收标准**

- 相同原文链接的新闻只保存一条。
- `canonical_url` 必须去除 `utm_*`、`fbclid` 等常见跟踪参数。
- 重复新闻不会重复触发全文抓取。
- 不同 `canonical_url` 的新闻可以正常进入全文抓取候选集合。

### 闭环流程 3.2：抓取新闻全文

**输入**

- 待抓取新闻原文链接

**输出**

- 原文全文内容
- 全文抓取状态

**运行流程**

1. 系统请求新闻原文链接。
2. 系统抽取页面中的正文内容。
3. 如果正文抽取成功，保存原文全文。
4. 如果正文抽取失败，使用 RSS 摘要作为内容兜底。
5. 系统不保存独立内容来源字段；是否使用兜底内容由 `content_full` 是否为空判断。
6. 如果原文全文和 RSS 摘要都不可用，当前新闻不得进入 API/UI 可展示集合。
7. 内容可用时，将数据库 `pipeline_state` 更新为 `fetched`。

**验收标准**

- 原文页面可访问且能抽取正文时，数据库保存原文全文。
- 原文页面不可访问或抽取失败时，数据库保存 RSS 摘要兜底内容。
- 每条进入翻译流程的新闻都有可用的原文内容或摘要内容。
- 内容降级优先级必须为：原文全文、RSS 摘要、不可展示。
- 内容抓取或摘要兜底成功后，数据库 `pipeline_state` 必须更新为 `fetched`。

## 功能 4：展示就绪与 LLM 中文翻译

### 闭环流程 4.1：保存高价值新闻为可展示候选

**输入**

- 去重后的高价值新闻
- 原文标题
- 原文链接
- RSS 摘要
- 原文全文或 RSS 摘要兜底内容
- 新闻价值评分

**输出**

- 可由 API 投影为 `ready` 的新闻记录

**运行流程**

1. 系统读取完成评分、去重和内容抓取的高价值新闻。
2. 系统保存原文标题、原文链接、来源、信息源发布时间和评分。
3. 系统保存原文全文或 RSS 摘要兜底内容。
4. 系统保持数据库 `pipeline_state = fetched`。
5. API 在中文字段不存在且未失败时投影为 `status = ready`。
6. 系统触发翻译流程，并用 `processing_log(stage = translate)` 记录处理结果。

**验收标准**

- 评分大于或等于 `60` 且内容可用的新闻会被 API 投影为 `ready`。
- 可展示新闻保留原文标题字段。
- 可展示新闻可以出现在主页面列表和 30 天高分榜单中。
- 翻译未完成不影响新闻展示。
- 新闻达到可展示条件后，数据库 `pipeline_state` 仍为 `fetched`。
- 数据库不得写入独立 `ready` 状态。

### 闭环流程 4.2：翻译新闻标题、摘要和正文

**输入**

- 原文标题
- 原文摘要
- 原文全文或 RSS 摘要兜底内容
- 新闻价值评分
- 新闻来源

**输出**

- 中文标题
- 中文摘要
- 中文正文
- 更新后的 API/UI 展示状态

**运行流程**

1. 系统读取 `pipeline_state = fetched` 且存在可用内容的新闻。
2. 系统按标准 JSON 格式将原文标题、摘要、正文、来源和评分发送给 LLM。
3. LLM 按标准 JSON 格式返回中文标题、中文摘要和中文正文；分类字段不得入库或暴露给 API/UI。
4. 系统校验返回字段不能为空。
5. 系统保存翻译结果。
6. 翻译成功只写入 `title_zh`、`summary_zh`、`content_zh`，API 投影为 `status = translated`。
7. 如果翻译失败，系统不写入部分中文字段，写入失败 `processing_log`，并设置 `has_translate_failed = 1` 供 API 投影为 `status = translation_failed`。

**验收标准**

- 翻译成功后，中文标题、中文摘要、中文正文均不为空。
- 如果全文抓取失败但 RSS 摘要可用，中文摘要和中文正文仍能基于 RSS 摘要生成。
- 翻译成功后，原文标题字段仍被保留。
- 翻译失败时，API/UI 投影状态必须为 `translation_failed`。
- 前端中文字段不得把未翻译英文内容伪装为中文返回。
- 翻译成功后，API/UI 投影状态必须为 `translated`。



## 功能 5：主页面新闻卡片

### 闭环流程 5.1：加载新闻卡片列表

**输入**

- 展示就绪、已翻译或翻译失败的新闻记录

**输出**

- 新闻卡片列表

**运行流程**

1. 用户打开主页面。
2. 前端请求新闻列表接口。
3. 后端按信息源发布时间倒序返回 API/UI 状态为 `ready`、`translated` 或 `translation_failed` 的新闻。
4. 前端渲染新闻卡片。
5. API/UI 状态为 `translated` 的卡片显示中文标题、中文摘要、来源、信息源发布时间和评分。
6. API/UI 状态为 `ready` 的卡片显示原文标题、来源、信息源发布时间、评分和“翻译中”状态，不显示英文摘要。
7. API/UI 状态为 `translation_failed` 的卡片显示原文标题、来源、信息源发布时间、评分和“翻译失败”状态，不显示英文摘要。
8. 系统自动为每个来源配置不同的卡片颜色。

**验收标准**

- 主页面能展示新闻卡片列表。
- 已完成翻译的卡片必须显示中文标题、中文摘要、来源、信息源发布时间和评分。
- 未完成翻译的卡片不得显示英文摘要，必须显示“翻译中”状态。
- 翻译失败的卡片不得显示英文摘要，必须显示“翻译失败”状态。
- 卡片摘要必须显示文本内容，不得显示原始 HTML 标签。

### 闭环流程 5.2：点击新闻卡片进入浏览页

**输入**

- 新闻卡片 ID

**输出**

- 新闻浏览页

**运行流程**

1. 用户点击新闻卡片标题。
2. 前端跳转到新闻浏览页路由。
3. 前端请求新闻详情接口。
4. 如果新闻 API/UI 状态为 `translated`，后端返回该新闻的中文详情数据。
5. 如果新闻 API/UI 状态为 `ready`，前端显示转圈等待动画，并轮询详情接口。
6. 如果新闻 API/UI 状态为 `translation_failed`，前端显示翻译失败提示和返回入口。
7. 翻译完成后，前端渲染新闻浏览页。

**验收标准**

- 点击卡片标题后进入站内浏览页。
- 浏览页显示的新闻 ID 与点击卡片一致。
- 浏览页不直接跳转原文站点。
- 浏览页显示中文内容。
- 浏览页显示的内容包含新闻标题，全文，来源，信息源发布时间。
- 翻译未完成时，浏览页显示等待动画，不显示英文全文。
- 翻译失败时，浏览页显示失败提示，不显示英文全文。

## 功能 6：30 天高分新闻榜单

### 闭环流程 6.1：生成 30 天高分榜单

**输入**

- 最近 30 天展示就绪、已翻译或翻译失败的新闻
- 新闻价值评分

**输出**

- 评分最高的前 10 条新闻

**运行流程**

1. 后端查询最近 30 天 API/UI 状态为 `ready`、`translated` 或 `translation_failed` 的新闻。
2. 后端按评分从高到低排序。
3. 如果评分相同，按信息源发布时间从新到旧排序。
4. 后端返回前 10 条新闻标题、ID、来源和评分。
5. 前端在主页面侧边栏展示榜单。
6. API/UI 状态为 `translated` 的新闻展示中文标题。
7. API/UI 状态为 `ready` 的新闻展示原文标题和“翻译中”标记。
8. API/UI 状态为 `translation_failed` 的新闻展示原文标题和“翻译失败”标记。

**验收标准**

- 榜单最多显示 10 条新闻。
- 榜单只包含最近 30 天内 API/UI 状态为 `ready`、`translated` 或 `translation_failed` 的新闻。
- 榜单按评分从高到低排序。
- 榜单中的每条新闻都保留原文标题字段。
- 榜单中的未完成或失败翻译新闻不得显示英文摘要或英文正文。

### 闭环流程 6.2：点击榜单标题进入浏览页

**输入**

- 榜单新闻 ID

**输出**

- 新闻浏览页

**运行流程**

1. 用户点击榜单中的新闻标题。
2. 前端跳转到新闻浏览页路由。
3. 前端请求新闻详情接口。
4. 如果新闻 API/UI 状态为 `translated`，后端返回该新闻的中文详情数据。
5. 如果新闻 API/UI 状态为 `ready`，前端显示转圈等待动画，并轮询详情接口。
6. 如果新闻 API/UI 状态为 `translation_failed`，前端显示翻译失败提示和返回入口。
7. 翻译完成后，前端渲染新闻浏览页。


**验收标准**

- 点击榜单标题后进入站内浏览页。
- 浏览页显示的新闻 ID 与榜单项一致。
- 浏览页显示的内容包含新闻标题，全文，来源，信息源发布时间。
- 翻译未完成时，浏览页显示等待动画，不显示英文全文。
- 翻译失败时，浏览页显示失败提示，不显示英文全文。

## 功能 7：新闻浏览页

### 闭环流程 7.1：展示中文全文

**输入**

- 新闻 ID

**输出**

- 新闻浏览页

**运行流程**

1. 用户从卡片或榜单点击新闻标题，前端根据路由中的新闻 ID 请求详情接口。
2. 后端查询新闻记录。
3. 如果新闻 API/UI 状态为 `translated`，后端返回中文标题、中文摘要、中文正文、原文标题、来源、评分和原文链接。
4. 如果新闻 API/UI 状态为 `ready`，后端返回当前展示状态。
5. 如果新闻 API/UI 状态为 `translation_failed`，后端返回翻译失败状态和原文链接，不返回英文全文。
6. 前端在翻译未完成时显示转圈等待动画。
7. 翻译完成后，前端渲染新闻浏览页。

**验收标准**

- 从卡片或榜单点击标题后进入站内新闻浏览页。
- 浏览页必须显示中文标题、中文正文。
- 浏览页必须保留并可展示原文标题字段。
- 浏览页必须显示来源、信息源发布时间和评分。
- 浏览页必须提供原文链接按钮。
- 翻译未完成时，浏览页必须显示等待动画，不得展示未翻译英文全文。
- 翻译失败时，浏览页必须显示失败提示，不得展示未翻译英文全文。

### 闭环流程 7.2：处理详情页异常状态

**输入**

- 不存在的新闻 ID
- 不可展示或不可访问的新闻记录

**输出**

- 明确的错误提示

**运行流程**

1. 前端请求新闻详情接口。
2. 后端判断新闻是否存在且可展示。
3. 如果不存在或不可展示，后端返回结构化错误。
4. 前端显示错误提示，并提供返回主页面入口。

**验收标准**

- 不存在的新闻 ID 返回 `404`。
- 前端显示可理解的错误提示。
- 错误状态下页面不出现空白页。

## 功能 8：系统状态、数据对象与 LLM 约束

### 闭环流程 8.1：统一数据库流程状态与 API/UI 展示状态

**输入**

- 新闻记录当前数据库 `pipeline_state`
- 评分、去重、全文抓取、可展示内容事实或翻译处理结果

**输出**

- 更新后的数据库 `pipeline_state` 或 API/UI `status` 投影

**运行流程**

1. RSS 条目首次入库时，数据库 `pipeline_state = raw`。
2. 评分完成后，数据库 `pipeline_state` 从 `raw` 更新为 `scored`。
3. 评分大于或等于 `60` 时，系统写入 `is_selected = 1`；评分小于 `60` 时写入 `is_selected = 0`。
4. 全文抓取或 RSS 摘要兜底内容可用后，数据库 `pipeline_state` 从 `scored` 更新为 `fetched`。
5. API/UI `status` 不入库，由 API 层按字段事实投影。
6. 中文字段完整存在时，API/UI `status = translated`。
7. 中文字段不完整且 `has_translate_failed = 1` 时，API/UI `status = translation_failed`。
8. 中文字段不完整且未失败时，API/UI `status = ready`。
9. 系统不维护独立翻译状态字段；翻译最终事实以中文字段、`has_translate_failed` 缓存和 `processing_log` 为准。

**验收标准**

- 数据库 `pipeline_state` 只允许取值：`raw`、`scored`、`fetched`。
- 新闻不得跳过中间状态直接从 `raw` 变为 `fetched`。
- 评分小于 `60` 的新闻停留在 `scored`，不会进入全文抓取。
- 翻译失败的新闻 API/UI 状态必须投影为 `translation_failed`。
- 主列表和榜单只展示 API/UI 状态为 `ready`、`translated` 或 `translation_failed` 的新闻。
- 数据库和 API 不得返回独立翻译状态字段。



### 闭环流程 8.2：最小核心数据对象入库

**输入**

- RSS 信息源
- RSS 新闻条目
- 抓取、评分、全文抓取和翻译处理结果

**输出**

- `source` 记录
- `news_item` 记录
- `processing_log` 记录

**运行流程**

1. 系统使用 `source` 统一保存默认和用户新增信息源。
2. 系统使用 `news_item` 保存新闻原文字段、中文字段、评分、去重键和数据库 `pipeline_state`。
3. 系统使用 `processing_log` 记录 `crawl`、`score`、`fetch`、`translate` 阶段的最小处理结果。
4. 每次定时抓取或手动刷新都写入对应的 `processing_log(stage = crawl)` 记录，并允许 pipeline-owned records 聚合开始时间、结束时间、成功数和失败信息作为内部证据。
5. crawl 处理读取所有启用且未删除的信息源、请求 RSS XML、解析 RSS 条目、创建或更新 `raw` 状态的 `news_item`。
6. score 处理对新条目执行标题和摘要评分，并为评分大于或等于 `60` 的非重复新闻写入 `is_selected = 1`。
7. fetch 处理只处理 `is_selected = 1` 且有可抓取内容的新闻。
8. translate 处理只处理 `pipeline_state = fetched` 且存在可用内容的新闻。
9. 各阶段只负责执行处理动作并记录执行结果，不直接跳过数据库流程状态规则。
10. 阶段失败后，系统在 `processing_log` 中记录阶段、关联对象、成功状态、时间和错误信息。

**验收标准**

- 数据库只包含 `source`、`news_item`、`processing_log` 三类 MVP 核心表。
- 删除或禁用 `source` 后，关联历史 `news_item` 仍保留；删除使用内部软删除事实，不要求物理删除 source 行。
- 每次手动刷新或定时抓取都能查询到对应的 `processing_log(stage = crawl)` 记录。
- crawl 记录必须表达处理成功或失败；详细聚合摘要不得要求新增任务表。
- crawl 处理不得直接保存原文全文内容。
- 处理阶段不得绕过数据库流程状态规则直接写入最终展示状态。
- 全文抓取失败或翻译失败时，失败原因必须记录在对应 `processing_log` 中。

### 闭环流程 8.3：标准化 LLM 调用

**输入**

- 评分请求 JSON
- 翻译请求 JSON

**输出**

- 评分响应 JSON
- 翻译响应 JSON
- LLM 调用状态

**运行流程**

1. 评分请求 JSON 必须包含 `title`、`summary`、`source`、`published_at` 和 `original_link`，其中 `published_at` 表示信息源发布时间。
2. 评分响应 JSON 必须包含 `score` 和 `reason`，其中 `score` 必须是 `0-100` 的数字。
3. 评分调用超时、返回非 JSON 或字段不合法时，系统最多重试 2 次。
4. 评分重试后仍失败时，系统不得写入无效 LLM 返回值；系统写入 fallback `score = 0`、`is_selected = 0`，将 `pipeline_state` 推进到 `scored`，写入失败 `processing_log(stage = score)`，且不进入全文抓取。
5. 翻译请求 JSON 必须包含 `original_title`、`original_summary`、`original_content`、`source` 和 `score`。
6. 翻译响应 JSON 必须包含 `title_zh`、`summary_zh` 和 `content_zh`；任何分类字段不得持久化或暴露。
7. 翻译调用超时、返回非 JSON 或字段不合法时，系统设置翻译失败事实，并记录失败原因。

**验收标准**

- 模拟 LLM 返回合法评分 JSON 时，系统能保存对应评分。
- 模拟 LLM 返回非法评分 JSON 时，系统最多重试 2 次；仍失败则由系统写入 fallback `score = 0`、`is_selected = 0` 和失败日志，不得写入无效 LLM 返回值。
- 模拟 LLM 返回合法翻译 JSON 时，系统能保存中文标题、中文摘要和中文正文。
- 模拟 LLM 返回非法翻译 JSON 时，系统不得写入伪造中文字段，API/UI 状态必须为 `translation_failed`。
- 模拟 LLM 超时或失败时，系统不得中断其他新闻处理任务。
